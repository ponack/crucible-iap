# Infrastructure as Code with Crucible — A Beginner's Introduction

Welcome. This page exists for people who are new to Infrastructure as Code (IaC) — or new to Terraform / OpenTofu — and want a one-page mental model before diving into the quickstart.

If you already know what `terraform apply` does and have used a state file before, you can skip to [`quickstart.md`](quickstart.md).

---

## What is Infrastructure as Code?

Infrastructure as Code means writing your infrastructure — servers, databases, DNS records, container clusters, even Slack channels — as text files that you check into Git. Instead of clicking buttons in a cloud console to create a VM, you describe the VM in a `.tf` file, and a tool reads that file and makes the cloud match.

The win is that your infrastructure becomes:

- **Reproducible** — spin up the same thing in dev, staging, and prod from the same code.
- **Reviewable** — changes go through pull requests like application code.
- **Versioned** — `git blame` tells you who changed the firewall rule and why.
- **Auditable** — every change is recorded in commit history.

Crucible is the layer that turns "git push" into a controlled, policy-checked, audited infrastructure change. You write the Terraform / OpenTofu / Pulumi code; Crucible runs it safely.

## The Terraform / OpenTofu workflow in one diagram

```text
   You write .tf files            ┌──────────────────────┐
        │                          │   tofu plan          │
        ▼                          │  ───────────────     │
  ┌──────────┐    git push    ┌────│ "I will create 3     │
  │  Editor  │ ───────────────►   │  resources, change   │
  └──────────┘                 │   │  1, destroy 0."      │
                               │   └──────────┬───────────┘
                               │              │
                               │              ▼  Review the plan
                               │   ┌──────────────────────┐
                               │   │  tofu apply          │
                               │   │  ───────────────     │
                               └──►│  Actually make the   │
                                   │  changes happen.     │
                                   └──────────────────────┘
```

Two distinct phases:

1. **Plan** — Terraform compares your code to the real world (and to a state file, see below) and produces a list of changes it *would* make. Nothing is changed yet.
2. **Apply** — Terraform executes the plan. Resources are created, modified, or destroyed.

The plan is the safety net. You read it, decide if it looks right, then approve. Crucible wraps this with an approval button, policy checks, audit logs, and PR comments — but the underlying loop is always `plan → review → apply`.

## State — the thing nobody mentions

Terraform needs to remember what it has already created. That memory is the **state file** (`terraform.tfstate`).

When Terraform runs `plan`, it loads the state to know "I created a VM with ID `vm-123` last time" — so it can detect that nothing has changed, or that someone deleted the VM outside Terraform (drift).

The state file is **the source of truth for what Terraform thinks exists**. It contains resource IDs, sometimes secrets, and the relationships between resources.

**Where it lives matters:**

- *Locally* — fine for learning, but a single laptop crash loses everything. Two people can't collaborate.
- *On a shared backend* — S3, GCS, a Terraform Cloud workspace, or **Crucible's built-in state backend**. This is the production answer.

Crucible includes a state backend out of the box — you do not need to set up S3 or anything else to get started. Every run reads from and writes to this backend automatically. Your state is encrypted at rest and lives on your infrastructure.

## Providers, resources, modules — the building blocks

A short glossary so the quickstart makes sense:

- **Provider** — a plugin that knows how to talk to a specific platform's API. `hashicorp/aws` knows AWS. `cloudflare/cloudflare` knows Cloudflare. You declare which providers you need at the top of your code.
- **Resource** — a single thing managed by a provider, like `aws_s3_bucket "logs"` or `cloudflare_record "www"`. The provider creates, updates, and destroys it.
- **Data source** — read-only lookup of something that exists (e.g. find an existing VPC by tag).
- **Variable** — input to your code. Pass different values for different environments.
- **Output** — value produced by your code (e.g. the IP address of a server you just created). Other stacks can read these.
- **Module** — a reusable bundle of resources, like a library. You can publish your own modules in [Crucible's built-in module registry](guides/provider-registry.md).

For the full vocabulary, see [`glossary.md`](glossary.md).

## Why use Crucible instead of running Terraform locally?

Running `tofu apply` on your laptop is fine for learning. For real infrastructure it is a problem:

| Problem with local Terraform | What Crucible does instead |
| --- | --- |
| State file lives on your laptop — can't collaborate | State stored on Crucible's encrypted backend; all team members run against the same state |
| You can apply without anyone reviewing | Every change goes through plan → approve → apply with required approvers |
| No audit trail of who applied what when | Every action recorded in an append-only audit log |
| Secrets sit in `terraform.tfvars` on disk | Stack env vars stored encrypted in Crucible's vault |
| No guardrails — anyone can `destroy` production | OPA policies block dangerous plans; per-stack approval gates |
| No history of past plans | Every plan artifact stored and viewable forever |
| Cloud credentials sit on your laptop | Workload Identity Federation — Crucible mints short-lived tokens, no static keys |
| Drift goes unnoticed until something breaks | Scheduled drift detection with alerts |
| Cost surprises | Infracost integration shows cost delta on every plan |

In short: Crucible adds the team, the gate, and the memory around Terraform.

## What if I already use Terraform Cloud or Spacelift?

You're not starting from scratch. See the [Spacelift migration guide](guides/spacelift-migration.md) for a concept mapping and state migration path. The same approach applies to Terraform Cloud: export state, point Crucible at it, redirect your `backend` block.

## What if I'm not sure where to host Crucible itself?

Crucible runs on a single host with `docker compose up`. For a small team this is enough — start there. When you outgrow it, the [operator guide](operator-guide.md) covers production deployments on AWS ECS, GCP Cloud Run, and Azure Container Apps with managed PostgreSQL and object storage.
